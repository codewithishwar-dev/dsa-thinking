# How to Approach a DSA Problem (Before Writing Code)

Most mistakes in DSA don’t happen in coding.

They happen before coding.

---

## Step 1: Understand the Constraints

- What is the size of input (n)?
- Can O(n²) pass?
- Is recursion safe?
- Is memory limited?

Constraints eliminate wrong approaches early.

---

## Step 2: Identify the Pattern

Ask:

- Is it Sliding Window?
- Two Pointers?
- Prefix/Suffix Sum?
- Binary Search on Answer?
- Graph traversal?
- DP with state transition?

Hard problems are often familiar patterns in disguise.

---

## Step 3: Define the State (If Needed)

For DP / complex problems:

- What does dp[i] represent?
- What decisions do I have?
- What is the recurrence relation?
- What is the base case?

Clarity in state = clarity in solution.

---

## Step 4: Think About Edge Cases

- Empty input
- Single element
- All same values
- Maximum constraints
- Negative numbers

Edge cases reveal logical gaps.

---

## Step 5: Optimize (Only If Needed)

First make it correct.
Then make it efficient.

Premature optimization creates complexity.

---

## Final Thought

Strong problem solvers don’t jump to code.

They model the problem first.

DSA maturity = structured thinking under constraints.
